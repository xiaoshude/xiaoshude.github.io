Vue 源码读后感

Posted on 2019-03-06 Edited on 2020-03-27

概括一下的话，就是不易读。

阅读 Vue 源码经常会有这样的感受：仿佛变量一会数字，一会字符串，各种条件覆写，让人琢磨不定。而且，函数内不同平台逻辑通过 if else 杂糅在一起，尽管有些地方使用 curry 化减轻了这种杂糅（比如著名的 patch 函数就是这样生成的）。 依然会让人感觉大脑“内存”不够用。

为什么不易读

其根本原因是 mixin 这种模式的使用。

Vue 的写法，让 divide & conquer 的学习策略失效，你必须在脑里时刻放着整个大 Vue，它的属性、方法、原型及每个方法受属性影响的分支逻辑，不然下一个方法你很可能看不懂，因为它的行为取决与上面一个超大的 context(Vue)。这里你也可以看到 mixin 的优点，它让信息的交换更便捷，弊端就是几乎没有哪个行为是 context free 的，耦合度大大上升。

这就是一个库设计的权衡：

  • 信息交换更便捷
  • 耦合度（复杂度）更低
    两个不可兼得。

mixin 在小型项目或局部是很好的，短平快。如果已经到 Vue 这种，mixin 会导致复杂度慢慢失控。

从背景理解

从Vue的起源看，这种设计选择就很容易理解了。
Vue 起源于尤大学习 Angular 的一个练手项目。所谓练手，当然怎么顺手怎么来啦，mixin 可谓对数据共享和交换带来极大便利。也正是这种起源，带来了 Vue 的复杂度瓶颈。

Vue 3 开始打破这种设计，在文档里你会看到一个很高频词：standalone，可以独立使用，也意味着每个功能不再耦合，这让整体复杂度迅速塌陷。

说到这，忍不住想插一腿 React。因为这种从背景考察的方式，也会让你迅速理解 React 设计选择，比如为什么用 setState，主动告知状态变动，而不像 Vue 自动搜集。

回过来，大多数复杂系统的设计，多会看到分层，比如操作系统、网络包的发送与解析等。这种设计和常说的管道模式有相似的感觉。不过说管道时，更多从数据流动的角度观察。各个功能解耦，但代价是，信息流动需要在不同格式之间倒腾。
不过相对于复杂度的降低来说，这些弊端是微不足道。
