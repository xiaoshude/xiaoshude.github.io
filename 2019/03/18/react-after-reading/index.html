React 读后感

Posted on 2019-03-18

在了解 React前，已经听到很多关于它的评价 —— 如何的具有开创性，比 Angular 多高效，比 Vue 多好用… 我通常对框架粉的言论比较审慎。

每个方案都是权衡的结果，即使一个框架平衡性做的足够好，各方面都不差，它也必然为这种通用性付出了代价 —— 放弃了特殊场景的效率。

熟悉 Vue 之后，看 React 并没有费太多力，基本看完文档，大概能想到如何实现的。React 和 Vue 有很多不同，如果细想他们各种出现的背景，为解决的问题，就很容易理解各自的选择。

背景

React 诞生在 Facebook，为大型应用设计，通用和兼容是它不得不考虑的。

  • 兼容，这个要求就直接决定了 React 不可能像 Vue 那样使用 Object.defineProperty 做响应。
  • 通用，服务端渲染、web 端渲染等，决定了 Vnode 的引入

基本 React 的弊端，都是为了实现这两个目的而付出的代价。

React 的选择

采用 setState，diff state 是否发生变动，然后标记组件 dirty。这个机制是不是很熟悉？对，就是 Angular 的脏检查。

为啥这个被骂惨了的 Angular 特性，在 React 成了优点呢，React 如何解决的？
其实 React 并没有解决这个问题，而是建议使用 immutable data 🤦。如果 Angular 使用 immutable data 避免深层 diff，也很快呀~

React 不用深层 diff 了，如何“精准”派发 DOM operations 呢？
其实深层 diff 还是需要，只不过被推迟到 patch 阶段，对 VNode 进行深层 diff，寻找 tree 的最小编辑距离。但是这个 n 三方的复杂度，让 React（包括 Vue）承受不起。

于是熟悉的一幕出现了：React 选择牺牲精确解，换得线性复杂度。这个代价表现在，DOM 的更新并不是精准的。而且如果不符合算法假设，代价会被放大。当然这个问题 Vue 也有。即使这样，随着组件嵌套过深，diff 依然很耗时。

React 为了缓解 VNode diff 占用主线程过久，导致无法响应用户操作，选择把 diff 计算过程打碎，使用 requestIdleCallback 分散。但是计算量并没有减少。

总的来说，React 有很多出色的地方，但这些特性在其他地方也能看到影子，非 React 创造，但 React 把这些很好的融合起来，并在设想的场景中达到较好的平衡。
