应用

1. form 组件的设计

在对比下两种写法：

左边：每个函数依赖的是另一个函数的【参数】，而参数和实现相关，所以间接依赖另一个函数的实现。
右边：依赖的是【接口】，只要是数字就可以。

Summary：

  • 当存在依赖，应该为这种依赖建立媒介 (接口、或协议)；
  • 依赖接口或者协议，而不是实现细节。

有个单词比较恰当：abstraction – 接口 / 协议

2. VScode

VScode 相对一般文本编辑器的亮点：

  • 代码理解（code understanding）
  • 调试 (debug)

Vscode 通过插件实现的代码理解，中间的桥梁就是 Language Server Protocol(LSP)，
一个基于 JSON 的文本协议，用于进程间通信。

最终达到什么效果呢：

  • VScode 里发生的每个动作，外面都能感知，并施加影响
  • 沟通的规则就落在这些协议里

Summary：

如何让库足够灵活且易扩展

  • 抽象内部关键动作，并给予外部通知及干预办法。
  • 为这种沟通建立完整规则（协议）

例2

Background: 下面的代码摘自 acr-cli 的API文档解析模块，模块本身很小而且功能不复杂。

context 是不是相当于一个媒介，只需要在关键节点记录数据，消费的时候读取自己需要的数据。
有点 in memory file 的感觉，把信息的记录和消费分离，其实大部分进程间沟通也可以理解为 in memory file。

但它的弊端也是很明显的，因为 context 的存在，基本每个函数都被耦合了，context.currentAction 变下记录的 key，就会把 warn 功能给 break。但是 context 的存在明显增加了信息交换的便利，而对于小型模块，对信息交互的需求往往强于解耦。

这里忍不住插下纯函数。
纯函数最近成了高频词，往往提到的好处是：便于调试(解耦的自然结果)。
不要忘记任何选择都是 tradeoff。纯函数也一样，它会带来了复杂度的上升（信息交换不便），而且带背景信息的函数更易于对现实建模。

应用

  • ​减少函数的参数
  • 历史记录影响当前动作的场景（上下文相关）

Acribus: 大多数辅助函数的参数减少都是通过 context 来做的。

schart:

  • background 单例，记录整个图表库的信息，同时规范信息结构，方便外面编程控制。
  • ChartBackground 每个实例记录一个图表的信息，同时规范信息结构，方便外面编程控制。

这里再总结下：

弊端：耦合度上升，函数将不在是纯函数；
建议：

  • 对外接口上，我偏好 context。因为这对现实能进行比较好的模拟：人的每个动作，都是有「语境」的，函数的行为也会根据之前的记录做调整，用起来更省心，但是把握不好很容易跑到对立面 —— 捉摸不定。

  • 有些功能的实现必须要求存在 context，比如修改、撤销这种操作，因为canvas是 context free的，也就是说，对之前的动作没有记忆（svg 不是，从这个角度看svg更高级点，贴近人的经验），必须应用自己记录，这也是canvas类库存在的意义 —— 提供一个有记忆的 canvas。

一个库怎样才算好用

和用户最近的是接口，所以如何设计接口直接关系易用性。

接口的风格可以大致分为：

  • 声明式：表达”要什么“，表现往往是配置。
  • 命令式：表达”要怎样“，往往表现为函数。

这里又是另一个权衡：
显然声明式是易用的，怎么做都被下层消化了，但是支持的功能和配置数量是正相关的，如果支持的功能很丰富，就会出现 ECharts 的现状。
命令式，意味着上层对下层有更大的控制能力（灵活），可以按照需求要求下层以某种方式处理。但无疑是繁琐的。

有没有办法简洁和灵活兼得呢？
Schart 常用功能配置完成，复杂功能编程控制，同时兼顾简洁和灵活。

好用的另一个指标

学习成本低。
如果新的东西越能迅速和以前的经验关联起来，学习的成本越低。

Angular 和 Vue 就是两个对比鲜明的例子。

总结

好用涉及的方面：

  • 上手时
    • 成本低
      • 新概念
      • API 个数和 API 的参数个数
  • 使用时
    • 简洁
    • 灵活

所以结束了吗

当你的库不能满足使用者的功能需求时，
使用者就转变成了开发者。
而这几乎总会发生。

这时就要考虑是否易扩展，代码结构是否清晰等。

  • 对使用者：关心的是好用
  • 对开发者：关心的是易维护、易扩展
