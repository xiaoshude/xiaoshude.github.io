说说 VNode

Posted on 2019-05-01

经常听到的一个说法：VNode 引入让 DOM 操作更高效。这个错误让我每次忍不住纠正。

VNode 并不是 Web 必须的，而是跨端必须的。而且 Web 还会因为 VNode 降低效率。

怎么理解这句话恁？

组件内数据变动时，是可以精确知道影响的具体DOM（在模版编译阶段），但是 Vue 丢掉了这个信息，放在了 VNode diff 的过程中，再次获得这个信息。而且 diff 还无法获得最小编辑距离。

其实不仅 Vue采用了 VNode，React 也采用了，当然可以解释为，他们都需要跨端，但是不是还有其他原因呢？

我们可以假设，如果跳过 VNode，把依赖收集的 watcher 改为直接发送 DOM 操作，这样会带来什么问题吗？
复杂度陡升，DOM 操作和数据变动耦合了。其实关键点就在这。

依然是一个权衡：

  • 执行效率高 - 往往意味着更短的流转路径、更少的信息倒腾
  • 复杂度低 - 相同信息往往要经历：专有格式 -> 通用格式 -> 专有格式的不断倒腾。

VNode 通过执行效率的稍微降低，带来解耦（复杂度降低）以及跨平台。

但是对于移动端，执行效率的优先级可能会升到最高。
数据变动到DOM操作派发，这个过程中大量的不必要计算，而且并不总是很快，不然 React 也不会推出异步渲染（Fiber），打碎 update 计算过程，但是一次 update 的计算量并没有减少。

这时候跳过 VNode 的 Web 专有框架显然是必要的。
